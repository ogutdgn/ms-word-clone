## 1. How to establish ground truth for EVERY feature

Before you judge or fix anything, understand the **real Word** behavior. Three sources, in order:

1. **Drive real Microsoft Word with computer-use.** The dev box has Word (`request_access`
   for `WINWORD`/`'Word'`). Click the actual ribbon control, open the actual dialog, trigger
   the actual contextual tab, run the actual flow end-to-end, and **screenshot each step**.
   Replicate *that* behavior — not a guess. (Canonical example: Insert ▸ Cover Page inserts a
   richly-formatted **building block** with live document-property fields and a distinct layout
   *per design* — NOT three `[Placeholder]` paragraphs. Our current Cover Page collapses all 6
   designs to the same stub. Go do the real flow and match it.)
2. **Deep-dive the web** (`WebSearch`/`WebFetch`): exact dialog fields, gallery contents, flow
   branches, keyboard shortcuts, and the **OOXML constructs** each produces (support.microsoft.com,
   learn.microsoft.com, ECMA-376). Capture every sub-flow and every hidden/contextual tab.
3. **Validate fidelity with the Word-COM oracle** (`scripts/oracle/*`, `validate-*-win.ps1`)
   for anything that writes OOXML — author a probe → `wordAPI.saveBytes` to `C:/tmp` → PID-safe
   validate script reads it back in real Word. This is the parity gate for every fidelity claim.

For ambiguous behaviors, **get a second read from Codex** (your teammate — §6): have it independently
describe the real-Word flow / OOXML and diff it against what you captured; resolve disagreements by
re-checking real Word.

> The reference build is **Word for Windows 16.0** (ADR-0006). "Par with Word" means the *flow,
> the dialog/contextual-tab UI, and the saved `.docx`* all match what real Word does.

## 4. Phase B — the fix loop (autonomous, no stopping)

```
for each section (Pages → … → E-signature):
  for each AUTO-FIX gap in the ledger, hardest-honest-value first:
    1. branch (short-lived local) off main
    2. research the real-Word flow (computer-use + web) if not already captured
    3. implement via the WC.PM bridge (NO-FORK default; minimal additive fork edit only if
       genuinely required — mark it `MS-WORD-CLONE FORK EDIT (0NN, user-authorized)`)
    4. write the regression test in scripts/test-suite-pm.js (FIRST, red → green)
    5. npm run build && the 4 gates (test:pm, test:smoke, test:roundtrip, test:bundle)
    6. Word-COM oracle for any OOXML/fidelity claim (author probe → saveBytes → validate-*-win.ps1)
    7. /code-review the diff (+ a Codex adversarial pass on tricky slices — §6); fix confirmed findings
    8. ff-merge to main, push, delete the branch; flip the ledger row to ✅; correct stale docs
    9. checkpoint (plan-tracking) and CONTINUE to the next gap — do not pause
  when a section's AUTO-FIX gaps are done, move to the next section
when all AUTO-FIX gaps are done: summarize the NEEDS-USER queue for me and stop
```

Batch trivially-related one-liners into a single `fix(insert)` commit; give substantial features
their own spec-kit slice (`specs/0NN-insert-*/` with a concise spec.md + a SPIKE that confirms
NO-FORK vs fork). Prefer **breadth of working flows** over deep rabbit holes — a feature that needs
a whole new subsystem (SmartArt diagram engine, live chart engine, OLE host) is NEEDS-USER, not AUTO.

## 5. Autonomy boundary

**AUTO-FIX — just do it, no asking** (clear correctness + bounded scope):
- Bugs where the dialog collects a value the bridge ignores (e.g. Date & Time "Update
  automatically" RB-050; gallery/design names dropped; Page-Number format/start-at/Remove).
- Missing flows that are well-defined and NO-FORK or a small additive fork edit (Drop Cap Options
  dialog; Header/Footer Remove + built-in gallery; Text Box gallery + draw; Cover Page's 6 distinct
  templates; Quick Parts field-code edit / Building Blocks Organizer; Equation Alt+= + Ink/Save
  wiring; Convert Text↔Table; cross-reference page resolution).
- Wiring an already-computed result that just isn't inserted (Chart: the dialog already builds the
  SVG — insert it as an image so the control stops lying).
- A **missing contextual tab** whose target object already exists (e.g. Shape Format once shapes
  insert; Picture/Table audits).
- Correcting false "works" claims in the docs.

**NEEDS-USER — queue it, keep going, surface at the end** (needs my decision, not just my labor):
- Anything requiring a **new subsystem / external runtime**: SmartArt diagram engine, live
  `c:chartSpace` chart engine, OLE/Office embedding (Object Create-New), 3D model viewer,
  Signature/E-sign provider, Online/Stock image + add-ins backends. (Decide: build vs keep honest
  stub vs **remove from ribbon** like the Home Dictate/Sensitivity/Add-ins/Reuse-Files removal.)
- A **non-additive or risky fork edit** to `src/renderer/core/superdoc-fork/`.
- Visual-design judgment calls or scope ambiguity where Word itself offers several behaviors.
- The Shapes engine: a real DrawingML `wps:sp` auto-shape path is large — SPIKE it first, then
  flag whether it's an AUTO slice or NEEDS-USER.

When in doubt, prefer making the control **honest** (do the real thing, or clearly degrade) over
leaving a full-looking picker that no-ops.

## 6. Work with Codex — your engineering partner

**You (Claude) own this end-to-end** — you drive real Word, write the ledger, design, implement,
test, oracle-validate, and land every change. **Codex is your teammate**, a second senior engineer
you actively pair with — not a fallback you only reach for when stuck. Bring it in through the
`codex:codex-rescue` agent (hand it the context + files; it runs in the shared runtime). Pull it in
proactively at these moments:

- **Design review before a substantial fix.** Share your fix plan / SPIKE findings and ask Codex to
  poke holes: is NO-FORK really achievable? is there a simpler bridge verb? which OOXML edge cases
  will the Word-COM oracle catch? Reconcile its take with yours *before* writing code.
- **Second opinion on ambiguous Word behavior.** For fuzzy flows (how Cover Page building-block
  fields recalc, how Shapes should round-trip as DrawingML `wps:sp`, what Header gallery entries
  Word ships), have Codex independently describe the real-Word flow / OOXML and **diff it against
  what you captured** — where you disagree, go re-check real Word with computer-use.
- **Root-cause partner when something fights you.** A failing gate, a converter round-trip mismatch,
  a paged-render quirk — hand Codex the symptom + relevant files and **compare diagnoses** instead
  of grinding solo.
- **Adversarial diff review** (on top of `/code-review`) for the trickier slices: ask Codex to
  review the change like a skeptical reviewer and try to break it.
- **Two-design bake-off.** When a feature has two plausible designs, let Codex sketch one while you
  sketch the other, then you (the owner) pick or merge the best.

**Rules of engagement:** you stay the owner and single source of truth — Codex advises, reviews, and
can draft, but *you* decide, integrate, test, and commit. **Never land Codex output unread**: verify
it against real Word + the 4 gates + the oracle exactly as you would your own. When a decision was
shaped by a Codex consult, note it briefly (commit body / ledger: "design reviewed with Codex").
Codex is a *partner, not a gate* — if it's unavailable or unhelpful on an item, proceed; never block
the loop waiting on it.

## 7. Project rules you MUST follow (non-negotiable — from CLAUDE.md / AGENTS.md)

- **The `WC.PM` bridge is the only document-write path.** No `window.prompt`. Don't hand-edit the
  generated `ribbon-data.js` / `icons-fluent.js` (change the generators).

  non-additive/risky fork change.

- **Gates before every merge** (after `npm run build`): `test:pm`, `test:smoke`, `test:roundtrip`,
  `test:bundle` — all green.
- **Word-COM oracle for every fidelity/OOXML claim.** PID-safe (never kill the user's Word window).
- **`/code-review` on each diff**, address confirmed findings, before landing.
- **Merge mode = ff-merge straight to `main`** (no PR), then push + delete the branch. Commit style:
  `type(scope): subject` + what/why body + explicit `git add <path>`, **no AI trailer** (scope =
  `insert` for features, or `commands`/`ribbon`/`docx`/`docs` as appropriate).
- **plan-tracking** at each checkpoint: append to `docs/plan/last-point.md`, tick
  `docs/plan/execution-map.md`.
